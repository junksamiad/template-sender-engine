# WhatsApp AI Chatbot - Frontend Documentation

## 1. Introduction

This document provides a detailed description of the frontend component of the WhatsApp AI chatbot system. The frontend serves as the interface through which different businesses can send data to initiate WhatsApp conversations with recipients. Each frontend is custom-tailored to specific business use cases, while adhering to a common payload structure for communication with the backend API.

## 2. General Frontend Architecture

### 2.1 Common Characteristics

All frontend implementations, regardless of use case, share the following characteristics:

- **Authentication**: Secure authentication mechanism using API keys
- **Payload Structure**: Standardized payload format with specific data objects
- **Response Handling**: Consistent handling of API responses and error conditions
- **Logging**: Comprehensive logging for auditing and debugging
- **User Experience**: Intuitive interfaces tailored to business workflows

### 2.2 Standard Payload Structure

All frontends must send a payload with the following structure:

```json
{
  "company_data": {
    "company_id": "string",
    "project_id": "string",
    "api_key": "string",
    "company_rep_1": "string"
  },
  "recipient_data": {
    "recipient_first_name": "string",
    "recipient_last_name": "string",
    "recipient_tel": "string",
    "recipient_email": "string",
    "comms_consent": boolean
  },
  "project_data": {
    // Varies by use case
  },
  "request_data": {
    "request_id": "string", // Unique identifier for the request
    "channel_method": "whatsapp", // currently only whatsapp supported but in future will include email and SMS
    "initial_request_timestamp": "string" // ISO 8601 format (YYYY-MM-DDTHH:MM:SS.sssZ)
  }
}
```

#### 2.2.1 Required Fields Explanation

- **request_id**: A UUID v4 generated by the frontend for each request. This ID must be unique for each request and is used for:
  - Idempotency (preventing duplicate processing)
  - Request tracing across the system
  - Correlation with logs and error reports
  - Replay attack prevention

- **initial_request_timestamp**: ISO 8601 formatted timestamp (e.g., "2023-06-15T14:30:45.123Z") indicating when the request was initiated. This is used for:
  - Request validity window enforcement (requests older than 5 minutes may be rejected)
  - Audit trail and compliance
  - Performance monitoring
  - Replay attack prevention

- **comms_consent**: Boolean value indicating whether the recipient has given consent to receive communications. This is used for:
  - Regulatory compliance (GDPR, CCPA, etc.)
  - Tracking explicit opt-in status
  - Ensuring ethical communication practices
  - Required for all communication channels

### 2.3 API Communication

The frontend communicates with the backend through REST API calls:

1. **Endpoint**: `https://api.example.com/v1/initiate-conversation`
2. **Method**: POST
3. **Headers**:
   - Content-Type: application/json
   - Authorization: Bearer {API_KEY}
4. **Response Handling**:
   - Success: 200 OK with returned response object
   - Error: Appropriate HTTP status codes with error details

### 2.4 Response Structure

The router will respond immediately with a status code and response body. The frontend must handle these appropriately:

#### 2.4.1 Success Response (200 OK)

```json
{
  "status": "success",
  "request_id": "same-as-sent-request-id",
  "message": "Request accepted and queued for processing",
  "queue_timestamp": "2023-06-15T14:30:45.789Z"
}
```

#### 2.4.2 Error Responses

**Authentication Error (401 Unauthorized)**
```json
{
  "status": "error",
  "error_code": "AUTH_ERROR",
  "message": "Invalid or missing API key",
  "request_id": "same-as-sent-request-id-if-available"
}
```

**Validation Error (400 Bad Request)**
```json
{
  "status": "error",
  "error_code": "VALIDATION_ERROR",
  "message": "Request validation failed",
  "request_id": "same-as-sent-request-id",
  "validation_errors": [
    {
      "field": "path.to.field",
      "message": "Specific error message"
    }
  ]
}
```

**Rate Limit Error (429 Too Many Requests)**
```json
{
  "status": "error",
  "error_code": "RATE_LIMIT_EXCEEDED",
  "message": "Rate limit exceeded",
  "request_id": "same-as-sent-request-id",
  "retry_after": 60 // Seconds until retry is allowed
}
```

**Server Error (500 Internal Server Error)**
```json
{
  "status": "error",
  "error_code": "SERVER_ERROR",
  "message": "An unexpected error occurred",
  "request_id": "same-as-sent-request-id",
  "reference_id": "internal-error-reference-id"
}
```

### 2.5 Frontend Implementation Requirements

#### 2.5.1 Request ID Generation

The frontend must generate a UUID v4 for each request:

```javascript
// Example implementation
function generateRequestId() {
  return 'xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx'.replace(/[xy]/g, function(c) {
    const r = Math.random() * 16 | 0;
    const v = c === 'x' ? r : (r & 0x3 | 0x8);
    return v.toString(16);
  });
}
```

#### 2.5.2 Timestamp Generation

The frontend must generate an ISO 8601 formatted timestamp:

```javascript
// Example implementation
function generateTimestamp() {
  return new Date().toISOString();
}
```

#### 2.5.3 Retry Logic

The frontend should implement exponential backoff for retries:

- Initial retry after 1 second
- Second retry after 2 seconds
- Third retry after 4 seconds
- Maximum of 3 retries for 4xx errors (except 429)
- For 429 errors, respect the retry_after value in the response
- For 5xx errors, retry up to 5 times with exponential backoff

#### 2.5.4 Monitoring and Status Tracking
The frontend does not need to implement status tracking or monitoring functionality. Instead:

- All message tracking and monitoring is handled by downstream channel engines
- A separate dashboard application will be provided for monitoring message status
- The dashboard will use the request_id to track messages across the system
- Frontend applications only need to store the request_id for reference purposes
- Status queries should be directed to the monitoring dashboard, not implemented in each frontend

This approach keeps frontend implementations lightweight while providing comprehensive monitoring through a centralized dashboard.

### 3.1 Overview

The Cucumber CV Analysis frontend is designed for recruitment agencies to facilitate initial candidate screening via WhatsApp. In reality, this front end will actually be a button inside an existing application which follows this context. The recruiter will have recalled a job role on their system (by jobID) which will have been applied for by a number of candidates. Their system (called the cv-analysis-engine) would have scored the CV's by a match critria, and provided reasons why the CV had been scored accordingly. Those reasons will include that a numer of job requirements in the advert do not are not evidenced in the candidate's CV. This results in the system providing a list of clarification points that the recruiter should query with the candidate. So there will be a button in the recruiters front end (called clarify CV) that when clicked, will send an api request to our application which will then initiate an AI whatsapp bot to contact the candidate and ask them to clarify the points flagged by the cv-analysis-engine. So in this use case, an actual front end concept is embedded in the button in the external system. So what we need to do here for testing purposes is replicate that button, and embed some example information into it so when clicked, it sends that information to our endpoint. It would be good on our testing page to have a single button, and then to have a hardcoded but editable payload sat in a window that we can edit. It would also be good to be able to set the API key in this test front end as well.

### 3.3 Testing Workflow

1. **Configuration Setup**: Tester configures API endpoint and authentication
2. **Payload Preparation**: Tester loads or edits the JSON payload to simulate specific scenarios
3. **Validation**: System validates the payload format before submission
   - Ensures request_id is a valid UUID v4
   - Ensures initial_request_timestamp is a valid ISO 8601 timestamp
   - Validates all required fields are present
4. **Submission**: Tester clicks the "Clarify CV" button to trigger the API call
5. **Response Inspection**: Tester reviews the API response and status
6. **Iteration**: Tester modifies the payload to test different scenarios and edge cases

### 3.5 Technical Implementation

#### Testing Frontend Technologies
- React.js for UI components
- Monaco Editor for JSON editing
- Axios for API communication
- Material-UI for design components
- React JSON View for response visualization
- LocalStorage for saving configurations and templates

#### Features
- **Persistent Configuration**: Saves API keys and endpoints in browser localStorage
- **Template Management**: Ability to save and load different test payloads
- **Request History**: Maintains a log of recent API calls and responses
- **Export/Import**: Functionality to export and import test configurations
- **Responsive Design**: Works on desktop and tablet devices for testing on the go
- **Auto-generation**: Buttons to auto-generate valid request_id and timestamp values
- **Validation**: Client-side validation of payload structure before submission

#### Security Considerations
- Warning about storing real API keys in localStorage
- Option to use session storage instead for sensitive keys
- Automatic clearing of credentials after configurable idle time
- No server-side storage of credentials or payloads
- Masking of sensitive data in logs and history
